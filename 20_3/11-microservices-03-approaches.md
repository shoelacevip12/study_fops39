## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

### Детальное описание стек-решений

#### Стек 1: GitLab (как коробочное решение в рамках одного продукта) - приоритет 1

**Состав и взаимодействие:**

- Единый продукт, предоставляющий `Git`-репозиторий, `CI`/`CD`, `Container Registry`, `Wiki`, `Issue Tracking`.
- `CI`/`CD` описывается в файле `.gitlab-ci.yml` в корне каждого репозитория.
- Поддерживает шаблоны `include` и `extends` для повторяющегося кода конфигураций.
- Сборки запускаются по событиям `push`/`merge request` или вручную по кнопке с параметрами.
- Агенты сборки `GitLab Runner` разворачиваются на собственных серверах `Docker`, `Kubernetes`, `shell`.
- Секреты хранятся в `CI`/`CD` Variables с маскированием и привязкой к окружениям.

**Соответствие требованиям:**

| Требование | Реализация в GitLab |
|---|---|
Облачная система | `Self-hosted` в приватном облаке или GitLab.com `SaaS` |
`Git` + репозиторий на сервис | Встроенный `Git`, неограниченное количество репозиториев |
Сборка по событию | `Webhook` на `push`, `merge` `request`, `tag` |
Сборка по кнопке | `Manual jobs` с параметрами `CI`/`CD` variables |
Настройки к каждой сборке | `Variables`, `environments`, `artifacts` |
Шаблоны конфигураций | `include:template`, `include:project`, `extends` |
Секреты | `CI`/`CD` Variables `masked` + `protected`, интеграция с `Vault` |
Несколько конфигураций | Разные `stages`, `parallel jobs`, `matrix builds` |
Кастомные шаги | Любые `shell`-команды, кастомные `Docker`-образы |
Свои Docker-образы | Кастомные `image` в `job`, `Container Registry` |
Свои агенты | `GitLab Runner` на `Docker`/`K8s`/`shell`/`SSH` |
Параллельные сборки | `Parallel jobs`, `parallel:matrix` |

#### Стек 2: Forgejo + Woodpecker CI - приоритет 2

**Состав и взаимодействие:**

- **Forgejo** - легковесный `Git`-репозиторий. Предоставляет `Git`-хостинг, `Web UI`, `Issue Tracking`.
- **Woodpecker CI** - лёгкий `CI`/`CD`-движок. Интегрируется с `Forgejo` через `OAuth2` и `webhook`.
- `CI`/`CD` описывается в файле `.woodpecker.yml` в корне репозитория.
- Поддерживает шаблоны через `depends_on`, `matrix`, `when`.
- Сборки запускаются по событиям `push`/`pull request` или вручную.
- Агенты `Woodpecker Agent` разворачиваются на собственных серверах `Docker`.
- Секреты: встроенные в `Woodpecker UI` или внешний `Mozilla SOPS` под `GitOps`.

**Соответствие требованиям:**

| Требование | Реализация в `Forgejo` + `Woodpecker` |
|---|---|
Облачная система | `Self-hosted` в приватном облаке |
Git + репозиторий на сервис | `Forgejo` - полноценный `Git`-хостинг |
Сборка по событию | `Webhook` из `Forgejo` в `Woodpecker` |
Сборка по кнопке | `Manual pipeline` в `Woodpecker UI` |

Шаблоны конфигураций | YAML-якоря, `depends_on`, `matrix` |
Секреты | `Woodpecker Secrets UI` + `Mozilla SOPS` |
Несколько конфигураций | `Matrix`-сборки, несколько `pipeline`-файлов |

Свои Docker-образы | Кастомные `image` в `step` |
Свои агенты | `Woodpecker Agent` на `Docker` |
Параллельные сборки | Параллельные `steps` и `matrix` |

### Обоснование выбора

В качестве решения для обеспечения процесса разработки предлагается **GitLab** (как единый продукт).

**Обоснование:**

1. **Полное соответствие всем требованиям** - `GitLab` покрывает все 13 требований из коробки, без необходимости интеграции дополнительных продуктов.

2. **Минимальные требования в компетенциях** - один продукт вместо двух-трёх. Не нужно настраивать интеграцию между `Git` и `CI`/`CD`, не нужно отдельное решение для секретов. Команда инженеров обслуживает одну систему.

3. **Простота поиска специалистов** - `GitLab` - самый популярный `self-hosted` `Git`-продукт в корпоративном секторе. Большое сообщество, документация, курсы.

4. **Хранение секретов** - встроенные `CI`/`CD` Variables с маскированием и защитой - самое простое и популярное решение. Для более строгих требований - интеграция с `HashiCorp Vault` без дополнительных компонентов.

5. **Гибкость** - `GitLab Runner` можно развернуть где угодно `Docker`, `Kubernetes`, `shell`, `SSH`. Поддержка кастомных `Docker`-образов для сборки.

6. **Forgejo + Woodpecker** - альтернатива для небольших команд с ограниченными ресурсами, но требует больше ручной настройки и отдельного решения для секретов.

## Задача 2: Логи

Предложите решение для обеспечения сбора и анализа логов сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:

- сбор логов в центральное хранилище со всех хостов, обслуживающих систему;
- минимальные требования к приложениям, сбор логов из stdout;
- гарантированная доставка логов до центрального хранилища;
- обеспечение поиска и фильтрации по записям логов;
- обеспечение пользовательского интерфейса с возможностью предоставления доступа разработчикам для поиска по записям логов;
- возможность дать ссылку на сохранённый поиск по записям логов.

### Детальное описание решений

#### Решение 1: Grafana Loki + Promtail + Grafana - приоритет 1

**Состав и взаимодействие:**

- **Promtail** - агент на каждом хосте. Собирает логи из `stdout` `Docker`-контейнеров и файлов. Добавляет метки job, instance, container_name. Отправляет в `Loki`.
- **Loki** - центральное хранилище логов. Индексирует только метки, а не содержимое логов, что делает его значительно легче `Elasticsearch`. Хранит логи в сжатом виде на объектном хранилище или локальном диске.
- **Grafana** - UI для поиска и фильтрации логов через `LogQL`. Поддерживает сохранённые запросы со ссылками.

**Соответствие требованиям:**

| Требование | Реализация |
|---|---|
Сбор в центральное хранилище | `Promtail` на каждом хосте -> `Loki` |
Сбор из `stdout` | `Promtail` автоматически собирает stdout `Docker`-контейнеров |
Гарантированная доставка | `Promtail` использует механизм `retry` и `batch`-отправку |
Поиск и фильтрация | `LogQL` в `Grafana` - фильтрация по меткам и содержимому |
UI для разработчиков | `Grafana` с многоуровневым доступом |
Ссылка на сохранённый поиск | Сохранённые запросы в `Grafana` с прямыми ссылками |

#### Решение 2: Vector + Elasticsearch + Kibana - приоритет 2

**Состав и взаимодействие:**

- **Vector** - агент на каждом хосте. Написан на `Rust`, потребляет минимально ресурсов. Собирает логи из stdout и файлов, может трансформировать и парсить `JSON`, добавлять поля. Отправляет в `Elasticsearch`.
- **Elasticsearch** - центральное хранилище с полнотекстовым поиском. Индексирует содержимое логов, поддерживает сложные запросы.
- **Kibana** - UI для поиска, фильтрации, визуализации. Поддерживает сохранённые запросы и ссылки на них.

**Соответствие требованиям:**

| Требование | Реализация |
|---|---|
Сбор в центральное хранилище | `Vector` на каждом хосте -> `Elasticsearch` |
Сбор из stdout | `Vector source`, `docker_logs` или `file` |
Гарантированная доставка | `Vector buffer `+ `acknowledgements` |
Поиск и фильтрация | Полнотекстовый поиск `Elasticsearch` + `Kibana Discover` |
UI для разработчиков | `Kibana` с пространствами и ролями |
Ссылка на сохранённый поиск | Saved Search в `Kibana` с прямыми ссылками |

#### Решение 3: ELK Stack (Elasticsearch + Logstash + Kibana) - приоритет 3

**Состав и взаимодействие:**

- **Logstash** - агент на каждом хосте или центральный. Собирает логи, трансформирует, обогащает. Богатая база плагинов на `input`, `filter`, `output`.
- **Elasticsearch** - центральное хранилище и поиск.
- **Kibana** - UI для поиска и визуализации.

**Соответствие требованиям:**

| Требование | Реализация |
|---|---|
Сбор в центральное хранилище | `Logstash` -> `Elasticsearch` |
Сбор из stdout | `Logstash` `input` `stdin`, `file`, `tcp` |
Гарантированная доставка | `Logstash` `persistent queue` |
Поиск и фильтрация | Полнотекстовый поиск `Elasticsearch` + `Kibana` |
UI для разработчиков | `Kibana` с пространствами и ролями |
Ссылка на сохранённый поиск | `Saved Search` в `Kibana` |

### Обоснование выбора

В качестве решения для сбора и анализа логов считаю оптимальным стек **Grafana Loki + Promtail + Grafana**.

**Обоснование:**

1. **Минимальные требования к приложениям** - `Promtail` собирает логи из `stdout` `Docker`-контейнеров автоматически. Приложениям не нужно ничего менять.

2. **Простота развёртывания** - все три компонента запускаются через `Docker Compose` из официальных образов. Не требует настройки `JVM`, кластеризации `Elasticsearch`, управления индексами.

3. **Низкие ресурсные затраты** - `Loki` не индексирует содержимое логов, только метки. Это даёт значительную экономию RAM и CPU по сравнению с `Elasticsearch`. Для небольших проектов достаточно 1 GB RAM на всё решение.

4. **Гарантированная доставка** - `Promtail` использует `batch`-отправку с механизмом `retry` при временных ошибках.

5. **Удобный UI** - С `Grafana` Разработчики получают доступ до визуального анализа логов с возможностью сохранения запросов и создания ссылок.

6. **Vector + Elasticsearch + Kibana** - хорош, если требуется полнотекстовый поиск по содержимому логов, но требует больше ресурсов и внимания к настройке `Elasticsearch`.

7. **ELK Stack** - наиболее зрелое решение, но самое тяжёлое и сложное в эксплуатации. Предпочтителен только для крупных проектов с опытной командой и высокими требованиями к обработке логов.

Компетенции: средние - нужно ориентироваться в кластеризации, репликации и retention. | Те же `Prometheus` `client libraries`, так как `VictoriaMetrics` совместима с `Prometheus` `exposition format`. Дополнительно - `MetricsQL` для более гибок для запросов к бизнес-метрикам. |

### Описание решений

#### Решение 1: Prometheus + Grafana - приоритет 1

**Состав и взаимодействие:**

- **Node Exporter** - агент на каждом хосте. Собирает метрики `CPU`, `RAM`, `HDD`, `Network`. Отдаёт их по `HTTP` на порту `9100`.
- **cAdvisor** - сбор метрик контейнеров `CPU`, `RAM`, `HDD`, `Network` для каждого контейнера.
- **Prometheus** - центральный сервер. Забирает `pull`-модель метрики с экспортёров по расписанию. Хранит в `TSDB`. Выполняет запросы `PromQL`. Отправляет алерты через `Alertmanager`.
- **Grafana** - UI для визуализации. Подключается к `Prometheus` как `datasource`. Позволяет создавать дашборды, делать `ad-hoc` запросы и выводить их на панель.

**Соответствие требованиям:**

| Требование | Реализация |
|---|---|
| Сбор метрик со всех хостов | `Node Exporter` на каждом хосте |
| Метрики ресурсов хостов `CPU`, `RAM`, `HDD`, `Network` | `Node Exporter` - стандартные метрики `node_cpu`, `node_memory`, `node_disk`, `node_network` |
| Метрики ресурсов сервисов `CPU`, `RAM`, `HDD`, `Network` | `cAdvisor` для контейнеров или встроенные /metrics эндпоинты сервисов |
| Специфичные метрики сервисов | `Prometheus` `client libraries` - кастомные метрики в коде сервисов |
| UI с запросами и агрегацией | `Grafana Explore` - `ad-hoc` запросы `PromQL` с агрегацией `sum`, `avg`, `rate` |
| UI с настраиваемыми панелями | `Grafana Dashboards` - гибкие панели, графики, таблицы, статус-панели |

**Бизнес-метрики:**
Для реализации бизнес-метрик в стек `Prometheus` + `Grafana` используются:

- **Prometheus client libraries** - официальные библиотеки для экспорта кастомных метрик:
  - `prometheus_client` `Python` - простая аннотация декораторами
  - `prometheus/client_golang` `Go` - регистрация счётчиков и гистограмм
  - `micrometer` `Java` - стандарт для `Spring Boot` приложений
  - `prom-client` `Node.js` - для `Express`/`Koa` приложений
- **Pushgateway** - для `batch`-задач и короткоживущих процессов, которые не могут быть опрошены `pull`-моделью
- **Примеры бизнес-метрик:**
  - `orders_total` - количество заказов с метками `status`, `payment_method`
  - `revenue_total` - выручка с метками `product_category`, `region`
  - `request_duration_seconds` - гистограмма времени ответа API
  - `active_users` - количество активных пользователей
- Все метрики автоматически попадают в `Prometheus` и доступны в `Grafana` через `PromQL`

#### Решение 2: VictoriaMetrics + Grafana - приоритет 2

**Состав и взаимодействие:**

- **vmagent** - лёгкий агент для сбора метрик, альтернатива `Prometheus`. Потребляет меньше ресурсов, поддерживает репликацию и фильтрацию.
- **VictoriaMetrics single-node или cluster** - хранилище метрик. Полностью совместимо с `Prometheus` `exposition format`. Поддерживает `PromQL` и расширенный `MetricsQL`.
- **Grafana** - UI для визуализации. Подключается к `VictoriaMetrics` как `Prometheus` `datasource`.
- **Экспортёры** - те же `Node Exporter`, `cAdvisor`, кастомные /metrics эндпоинты.

**Соответствие требованиям:**

| Требование | Реализация |
|---|---|
| Сбор метрик со всех хостов | `vmagent` + `Node Exporter` |
| Метрики ресурсов хостов | `Node Exporter` |
| Метрики ресурсов сервисов | `cAdvisor` + кастомные /metrics |
| Специфичные метрики сервисов | `Prometheus` `client libraries` |
| UI с запросами и агрегацией | `Grafana` `Explore` с `PromQL`/`MetricsQL` |
| UI с настраиваемыми панелями | `Grafana Dashboards` |

**Бизнес-метрики:**
Полностью совместимо с подходом `Prometheus`:

- Те же **Prometheus client libraries** для всех языков
- **MetricsQL** - расширенный язык запросов, упрощающий расчёт бизнес-показателей:
  - `rollup_rate` - скользящее среднее для сглаживания выбросов
  - `rollup_share` - доля от общего значения
  - `default_rollup` - заполнение пропусков
- **vmbackup** - встроенное резервное копирование бизнес-метрик в S3

### Обоснование выбора

В качестве решения для мониторинга по простоте предпочтителен **Prometheus + Grafana**.

**Обоснование:**

1. **Полное соответствие требованиям** - `Prometheus` + `Node Exporter` + `cAdvisor` покрывают все требования по сбору метрик хостов и сервисов. `Grafana` предоставляет UI для запросов и дашбордов.

2. **Простота развёртывания** - все компоненты запускаются через `Docker Compose`. `Prometheus` настраивается одним `YAML`-файлом. `Node Exporter` и `cAdvisor` - стандартные `Docker`-образы.

3. **Минимальные компетенции** - Prometheus - стандарт де-факто в микросервисной архитектуре. `PromQL` легко изучить. Огромное сообщество и готовые дашборды.

4. **Бизнес-метрики** - `Prometheus` `client libraries` позволяют легко добавлять кастомные метрики в код сервисов. Все метрики автоматически доступны в `Grafana`.

5. **VictoriaMetrics** - Выбор для крупных проектов с высокими нагрузками, где требуется более эффективное хранение и масштабирование, но требует более глубоких компетенций.

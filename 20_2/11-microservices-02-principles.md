## Задача 1: API Gateway

Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.

Решение должно соответствовать следующим требованиям:

- маршрутизация запросов к нужному сервису на основе конфигурации,
- возможность проверки аутентификационной информации в запросах,
- обеспечение терминации HTTPS.

**NGINX** | `Веб-сервер` и `обратный прокси`. Конфигурация через файлы nginx.conf. Большое сообщество и экосистема доступных модулей. ### Обоснование выбора

В качестве решения для реализации API Gateway более предпочтителен **NGINX**.

**Обоснование:**

1. **Прост в эксплуатации** - `NGINX` имеет наименьший порог входа. Конфигурация описывается в одном файле, для базовой реализации не требует внешних зависимостей, таких как СУБД и дополнительных сервисов.
2. **Универсален** - NGINX одинаково хорошо подходит как для небольшого проекта, так и для корпоративной инфраструктуры. Проще масштабируется горизонтально через DNS-балансировку или в связке с `HAProxy`.
3. **Подходит под требования:**
   - Маршрутизация запросов - блоки json кода `location` и `upstream` в nginx.conf.
   - Проверка аутентификации - как пример, модуль `ngx_http_auth_request_module` для subrequest к сервису security или `lua-nginx-module`.
   - Терминация HTTPS - стандартная настройка SSL-сертификатов.
4. **Информативная доступность** - огромное сообщество, множество готовых модулей и примеров конфигурации.
5. **Довольно производителен** - событийно-ориентированная архитектура ПО обеспечивает высокую пропускную способность при низком потреблении ресурсов.

## Задача 2: Брокер сообщений

Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.

Решение должно соответствовать следующим требованиям:

- поддержка кластеризации для обеспечения надёжности,
- хранение сообщений на диске в процессе доставки,
- высокая скорость работы,
- поддержка различных форматов сообщений,
- разделение прав доступа к различным потокам сообщений,
- простота эксплуатации.

### Сравнительная таблица брокеров сообщений

| Решение | Описание | Приоритет по простоте эксплуатации | Сложности в эксплуатации | Целесообразность |
|---------|----------|-----------------------------------|----------------------|------------------|
**Redis Streams** | `in-memory` хранилище с поддержкой очередей через службу `Redis Streams`. Поддерживает `Pub`\\`Sub`, групп `consumers`, может сохранять данные на диск(`persistence`) через механизмы `RDB(Redis DataBase)`\\`AOF(Append-Only File)`. | 1 самый простой | Низкая. Один бинарный-исполняемый файл\служба, конфигурируется через текстовый config-файл. Не требует обязательных внешних зависимостей. Обновление - в рамках пакета установки. Кластеризуется отдельным пакетом программы `Redis Cluster` и настраивается отдельно. | Подходит для небольших и средних проектов, в особенности где есть совместимость или уже используется `Redis`. Среди брокеров сообщений по производительности при интенсивных нагрузках несильно уступает `RabbitMQ` и `Kafka`. |
**RabbitMQ** | Самый старый и проверенный временем брокер сообщений с поддержкой протоколов: `AMQP 0-9-1(Advanced Message Queuing Protocol версии 0-9-1)`, `MQTT(message queuing telemetry transport)`, `STOMP (Streaming Text Orientated Messaging Protocol)`. Есть поддержка в кластеризацию, но требует акцентированного внимания к учету сетевых задержек. При обновление требует остановки\перезапуску кластера. | Универсален для большинства проектов. Подходит как для небольшого сервиса, так и для корпоративного использования. Сбалансирован по соотношению функциональности и сложности в эксплуатации. |
**NATS JetStream** | Современный брокер на базе `NATS core(Neural Autonomic Transport System)`. `JetStream` дает возможность: сохранять данные на диск, выполнить доставку в режиме `exactly-once`, использовать `key-value` хранилища. Минималистичный дизайн управления на уровнях блоков кода `jetstream`: Server-Level, Account-Level и Streams. | 3 | Средняя. Проще в эксплуатации чем `RabbitMQ`, но меньше доступной открытой информации использования ПО в сети и документации. | Как и прочие, для высоконагруженных систем. Для малых проектов может быть избыточен. Подходящее решение для компании с микросервисной средой на `Go`\\`NATS`. |
**Apache Kafka** | Распределённая платформа потоковой обработки данных. Высочайшая пропускная способность, хранение на диске, лог-ориентированная архитектура. Поддержка `Kafka Connect`, `Streams`, `KSQL(Kafka SQL)`. | 4 самый сложный | Высокая. Требует `ZooKeeper` или `KRaft`, состоит из нескольких обязательных компонентов. При масштабном обновлении требует миграции и перезапуска кластера. | Предпочтителен для для крупных компаний под высоконагруженные системы. Избыточен для небольших проектов. |

### Обоснование выбора

В качестве решения для реализации брокера сообщений предпочтителен **RabbitMQ**.

**Обоснование:**

1. **Соответствие требованиям:**
   - Кластеризация - встроенная поддержка кластеров и очередей с зеркалированием.
   - Хранение на диске - все сообщения сохраняются на диск при настройке `persistent` доставки.
   - Высокая скорость - до десятков тысяч сообщений в секунду на одном узле.
   - Различные форматы - поддержка `AMQP`, `MQTT`, `STOMP`, `HTTP`; сообщения передаются в бинарном виде, поддерживаемые форматы для приложений под `JSON`, `XML`, `Protobuf`.
   - Разделение прав - встроенные vhosts, users, permissions, TLS-аутентификация.
   - Простота эксплуатации - вариативн от методов: `management UI`, `HTTP API`, готовые `Helm-чарты` или через `Docker-образы`.

2. **Зрелость проекта** - `RabbitMQ` существует с 2007 года, огромное сообщество, множество клиентских библиотек для многих языков программирования.

3. **Оптимальный баланс** - `RabbitMQ` занимает золотую середину между простотой `Redis Streams` и сложностью `Kafka`, предоставляя все необходимые функции для типовой микросервисной архитектуры.

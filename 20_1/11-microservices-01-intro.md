# Домашнее задание к занятию «`Введение в микросервисы`» `Скворцов Денис`

## Задача

Руководство крупного интернет-магазина, у которого постоянно растёт пользовательская база и количество заказов, рассматривает возможность переделки своей внутренней   ИТ-системы на основе микросервисов.

Вас пригласили в качестве консультанта для оценки целесообразности перехода на микросервисную архитектуру.

Опишите, какие выгоды может получить компания от перехода на микросервисную архитектуру и какие проблемы нужно решить в первую очередь.

---

Предполагаемое Текущее состояние ИТ-системы магазина:

- **Монолитное приложение**, включающее:
  - каталог товаров,
  - корзину,
  - оформление заказов,
  - платежи,
  - управление пользователями,
  - админ-панель,
  - систему рекомендаций,
  - поиск,
  - логистику/доставку.
- **Единая база данных**.
- **Один цикл релиза** - любое изменение требует переразвёртывания\обновления всего приложения.
- **Масштабирование "вертикальное"** - увеличение ресурсов одного сервера или клонирование всего монолита целиком.

## Выгоды от перехода на микросервисную архитектуру

### `Масштабируемость`

> Экономия ресурсов - масштабируется только то, что действительно нужно

| Компонент | Проблема монолита | Решение в микросервисах |
|---|---|---|
| Каталог товаров | Масштабируется вместе с платёжным модулем | Можно масштабировать независимо |
| Корзина / Оформление заказов | Периодические пиковые нагрузки при распродажах требуют масштабирования всего монолита | Масштабируется только сервис заказов |
| Поиск | Тяжёлые поисковые запросы нагружают всё приложение | Выделенный поисковый сервис масштабируется отдельно |

### `Независимость разработки и развёртывания`

> Разные команды могут работать над разными сервисами параллельно, а каждый сервис может применять изменения по своему графику

- Команда для работы с каталогом - над фильтрацией и поиском.
- Команда для работы  платежами - над интеграцией нового платёжного шлюза.
- Команда для работы с логистикой - над интеграцией с курьерскими службами.

### `Технологическая гибкость`

> Разработчики могут выбирать технологии для своих сервисов по своему усмотрению

Как пример стека:

- Поиск - **Elasticsearch / Meilisearch**.
- Рекомендации - **Python + ML-библиотеки**.
- Платежи - **Java / Go** для надёжности и производительности.
- Фронтенд - **Next.js / Nuxt** с отдельным бэкендом для каждого раздела (BFF).
- Возможность **поэтапной миграции** без переписывания всего сразу.

### `Отказоустойчивость`

> Отказоустойчивость - высокая доступность сервисов, что не должны блокировать работу других сервисов

- **Изоляция сбоев:** если один сервис недоступен, другие могут работать.
- **Graceful degradation:** если один сервис недоступен, другие могут работать, но с ограниченными возможностями.
- **Circuit Breaker:** при перегрузке одного сервиса трафик перераспределяется, не допуская каскадного отказа.

### `Простота поддержки и понимания кода`

> Простота поддержки и понимания кода среди разработчиков и архитекторов микросервисов

- Каждый микросервис - **относительно не большая база кода** которую легко понять как архитектору микросервиса и так разработчику.
- Понятные **границы ответственности**.
- Упрощение для вливания в процесс новых сотрудников.

### `Оптимизация затрат`

> Все За и Против, что внедрение микросервисной архитектуры сокращает Общие затраты на обслуживание и поддержку.

- **Точное выделение ресурсов:** сервис с низкой нагрузкой, например, админ-панель, может работать на минимальных ресурсах.
- **Использование Spot-инстансов** для нечувствительных к задержкам сервисов, таких как фоновые задачи или обработка изображений.

## Проблемы, которые нужно решить в первую очередь

### `Реструктурирование ИТ-команды`

> "Системы проектируются так, как организованы команды, которые их создают."

**Что делать:**

- Реорганизовать ИТ-команды по **принципу "команда на сервис\группу сервисов"**.
- Каждой команде следует адаптироваться под **полный цикл**: разработка, тестирование, деплой, мониторинг, поддержка.
- Придерживаться практик CI/CD и IaC.
- Старшие специалисты из ИТ-подразделения, что первоначально развертывали ИТ-систему магазина, привыкли к монолиту - нужны **тренинги и менторство**.
- **Постепенное** внедрение : начать с одного-двух сервисов с низким риском. Например, сервис уведомлений или сервис отзывов.

### `Мониторинг, Логирование, Трейсинг`

> Проблема: в монолите - один лог-файл и один APM-агент. В микросервисах - десятки процессов.

**Пример стеков Решения:**

| Решение для | Инструменты | Что даст |
|---|---|---|
| **Logging** | ELK / Loki + Grafana | Централизованный сбор логов со всех сервисов |
| **Metrics** | Prometheus + Grafana | Метрики производительности и статистики использования ресурсов |
| **Tracing** | Jaeger / Zipkin / OpenTelemetry | Сквозная трассировка запроса через все сервисы |

### `Сетевое взаимодействие и задержки`

> Проблема: внутри монолита вызов метода - это наносекунды. Между микросервисами - HTTP/gRPC с задержками 1–50ms.

**Решение:**

- Использовать **gRPC** вместо REST для внутренних сервисов.
- **Асинхронное взаимодействие** через **Message Broker** (Kafka / RabbitMQ / NATS) для событий, не требующих мгновенного ответа.
- **Кэширование** (Redis / Memcached) для часто запрашиваемых данных.
- **BFF** - агрегация данных на стороне API Gateway.

### `Приоритет на очередь миграции в микросервисную архитектуру`

> Обозначить первые сервисы на отделение от монолита. От простого к сложному.

**Решение:**

- **Миграция на микросервисы** с **приоритетом на очередь**:
  - **Уведомления (Email/SMS/Push)**
  - **Аутентификация / Auth**
  - **Каталог товаров**
  - **Поиск**
  - **Корзина**
  - **Заказы / сервисов**

| Приоритет | Сервис | Причина |
|---|---|---|
| 1 | **Уведомления (Email/SMS/Push)** | Минимальная связанность, легко изолировать |
| 2 | **Аутентификация / Auth** | Критический, но чёткие границы |
| 3 | **Каталог товаров** | Высокая нагрузка, нужна независимая масштабируемость |
| 4 | **Поиск** | Тяжёлые запросы, отдельный стек (Elasticsearch) |
| 5 | **Корзина** | Требует высокой производительности |
| 6 | **Заказы / Платежи** | Самые сложные — Saga, распределённые транзакции |
| 7 | **Логистика / Доставка** | Интеграция с внешними системами |
